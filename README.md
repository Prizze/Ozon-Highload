## 1. Тема, аудитория и функционал

### Тема

**Ozon** - один из крупнейших российских маркетплейсов, где тысячи продавцов предлагают широкий ассортимент товаров от электроники до продуктов питания.

### Функционал MVP

1. Профиль пользователя
    - Регистрация
    - Авторизация
    - Создание, редактирование, удаление товара
2. Товары
    - Поиск товаров
    - Фильтры и категории товаров
    - Карточка товара с описанием
3. Лента с рекомендуемыми товарами
    - Главная страница
    - Рекомендации в карточке товара
4. Отзывы
    - Создание, редактирование, удаление отзыва на купленный товар
    - Просмотр рейтинга товара
5. Корзина товаров
    - Добавление, удаление товара. Изменение количества товара
    - Подсчет итоговой суммы
6. Заказ
    - Оформление заказа
    - Выбор адреса и способа доставки
    - Просмотр статуса заказа
    - Просмотр истории заказов
    - Выбор способа оплаты заказа

### Исходные данные

|:- | :- |

Данные из веб-приложения Ozon:
- При поиске товара прогружается 12 товаров за раз.
- При просмотре ленты рекомендаций прогружается 10 товаров за раз.
- При просмотре отзывов прогружается по 8 отзывов за раз.

### Продуктовые метрики

#### Рассчитаем среднее количество действий пользователя по типам в день

Заказы:
- Учитывая, что в среднем пользователь делает 29.9 заказов в год, то в день он делает $\frac{29.9}{365} = \mathbf{0.082}$ заказа
- Предположение: На каждый заказ пользовтель заходит 3 раза в историю заказов: 1 раз при оплате, второй раз при доставке и еще раз для проверки во время его доставки. Тогда просмотр истории заказов в день: $0.082 * 3 = \mathbf{0.246}$

Поиск:

- Если в месяц происходит 1.18B поисковых запросов, то количество запросов на человека в день: $\frac{1.18 * 10^9}{30*DAU} = \frac{1.18 * 10^9}{30 * 22 * 10^6} = \mathbf{1.78}$

Корзина:

- Конверсия добавления товара в корзину из поиска в среднем 15.8% (взято с аналитики поиска [Ozon Seller](https://seller.ozon.ru/app/analytics/what-to-sell/all-queries) [^7]. Значение рассчитывалось как среднее по первым 50 типам запроса). Тогда количество добавлений товаров в корзину: $1.78 * 0.158 = \mathbf{0.28}$
- Предположение: пользователь смотрит корзину только перед заказом, тогда возьмем количество просмотров корзины равным количеству заказов - $\mathbf{0.082}$

Товары:
- Исходя из данных аналитики поиска с Ozon Seller (аналитика поисковых запросов)[^7] в среднем за поиск показывается 79 товаров. Учитывая, что прогружается сразу 12 товаров, то количество подгрузок товара при поиске в день: $\frac{79}{12} * 1,78 = \mathbf{11.7} $
- По данным Ozon Seller (аналитика популярных товаров) [^9] в среднем на 544372 просмотра товаров в поиске открывается 183401 карточка. То есть на $\frac{544372}{12} = 45364$ прогрузки товаров в поиске просматривается 183401 карточка. Тогда на каждую подгрузку товаров открыватеся $\approx 4$ карточки. Количество просмотренных карточек товара в день: $11.7 * 4 = \mathbf{46.8}$

Лента:
- В среднем пользователь заходит $\frac{DAU * 30}{MAU} = \frac{22M * 30}{60.5M} = 10.9$ раз в месяц. Тогда в день он заходит $\frac{10.9}{30} \approx \frac{1}{3}$ раз. Количество показов товаров в ленте возьмем равным количеству показов товаров при поиске, умноженным на 4: $79 * 4 = 316$. Тогда количество подгрузок ленты: $\frac{316}{10} * \frac{1}{3} = \mathbf{10.5}$


Отзывы:
- Отзывы к заказам оставляет только 10% покупателей [^8]. Тогда количество оставленных отзывов в день: $0.082 * 0.1 = \mathbf{0.0082}$
- При просмотре товара 95% пользователей читают отзывы [^8]. Предположение: Пользователь просматрвивает 16 отзывов при просмотре товара, то есть отзывы подгружаются два раза; пользователь читает отзывы на 3 самых лучших варианта. Также в корзину в среднем переходит 12.36% товаров после просмотра карточки товара[^9], то есть какой-то из трех лучших. Тогда количество подгрузок отзывов в день: $46.8 * 3 * 0.126 = \mathbf{17.6}$


Продавцы:
- Предположение: Каждый продавец выставляет 0.5 товара в день, тогда в месяц получается $600K * 0.5 * 30 = 9M$, тогда в год будет $108M$  новых товаров. Учитывая, что всего товаров на момент 2023 г. было 370M [^10], можно сказать, что прирост количества товаров реалистичный.

Профиль:
- Во втором квартале 2025 г. активных покупателей было 60.5M, в первом квартале - 58.1M. Прирост за три месяца: $60.5M - 58.1M = 2.4M$. Значит в день регистрируется примерно $\frac{2.4 * 10^6}{90} = \mathbf{26K}$ людей
- Пусть повторных автоизаций будет примерно как и Регистраций, так как обычно пользователь редко выходит из аккаунта.

#### Рассчитаем сетевой трафик

 Для расчета будем использовать уже посчитанное раннее среднее количество действий пользователя по типам в день

Подбор товаров в поиске:
- Запрос весит 18 Кб. Также подгружается 12 картинок (по одной на каждый товар). Тогда общий трафик: 18 Кб + 12 * 12 КБ = 18 Кб запрос и 144 Кб на медиа
- За сутки: запрос - $\frac{11.7 * 18 * 22 * 10^6}{1024*1024} = 4418\ Гбайт/сутки$, медиа: $\frac{11.7 * 144 * 22 * 10^6}{1024*1024} = 35348\ Гбайт/сутки$

Карточка товара:
- На одну карточку товара - 36 Кб и ~8 картинок. Общий трафик: 36 Кб + 8 * 12 Кб = 36 Кб запрос и 96 Кб медиа
- За сутки: запрос - $\frac{46.8 * 36 * 22 * 10^6}{1024*1024} = 35348\ Гбайт/сутки$, медиа: $\frac{46.8 * 96 * 22 * 10^6}{1024*1024} = 94262\ Гбайт/сутки$

Поиск:
- На один запрос поиска нужно $1.1\ Кб$
- За сутки: $\frac{1.78 * 1.1 * 22 * 10^6}{1024*1024} = 41\ Гбайт/сутки$

Подгрузка отзывов:
- Для загрузки отзывов необходимо $17.9\ Кб$.
- За сутки: $\frac{17.6 * 17.9 * 22 * 10^6}{1024*1024} = 6609\ Гбайт/сутки$

Подгрузка товаров в ленте рекомандаций:
- Один запрос весит 15 Кб и 10 картинок. Общий трафик: 15 Кб + 10 * 12 = 15 Кб запрос и 120 Кб медиа

## 3.Глобальная балансировка нагрузки

### Функциональное разбиение по доменам
- **ozon.ru** - api
- **ir.ozone.ru** - медиа-контент

### Расположения ДЦ

Исходя из аналитики аудитория из п. 1, можно сказать, что почти весь трафик приходится на Россию. Значит, логично разместить ДЦ на территории РФ. Сейчас у Ozon 4 ДЦ [^1], предположим где они могли бы находиться. Исходя из местоположения крупных ЦОД [^11] и распределения населения в России[^12], можно выделить 4 хороших места для ДЦ:
- Москва (37% нагрузки)
- Казань (19% нагрузки) 
- Новосибирск (25% нагрузки)
- Ростов-на-Дону (19% нагрузки)

### Схема балансировки

#### DNS балансировка
Для API будем использовать Geo-based DNS + Latency-based DNS. Также можно использовать health-check ДЦ. Так учитывается распределение пользователей по Федеральным округам, но с учетом задержки запроса и работоспособности ДЦ. При отключении одного ДЦ, например в Москве, трафик можно распределять между Казанью и Ростовом-на-Дону с помощью Round-Robin.

#### Anycast балансировка
Для разадчи картинок будем использовать CDN. Для этого с каждым ДЦ будут связаны 2-3 POP. Они принимают запрос на картинку, в случае ее наличия в кеше возвращают ответ, иначе делают запрос на ДЦ и кешируют. Все эти POP имеют один IP, но с помощью BGP Anycast трафик перенаправлятся в ближайший. Это также позволяет повысить отказоустойчивость, так как в случае выхода из строя одного POP, перестроится сеть и BGP просто будет отправлять запрос на другой POP.

## 4. Локальная балансировка нагрузки

### Балансировка входящих запросов

- #### L7-балансировка - NGINX
    - На этом этапе происходит SSL-терминация, то есть завершается HTTPS-сессия и начинается общение по HTTP.
    - Дальше идет распределение в зависимости от домена (api или медиа).
    - Для api запрос маршрутизируется по URL
    - Для распределения нагрузки используется least connections и health-checks для проверки работоспособности бэкенда
- #### Kubernetes
    Для запросов на API-сервисы
    - L7 балансировщик передает запрос на ingress-nginx внутри кластера
    - Kubernetes выбирает нужный сервис и под
- #### Бэкенд
    - В контейнере запущено само приложение
    - Тут происходит бизнес-логика
    - Обращения к кешам или БД, запросы к другим сервисам, рвбота с очередями

### Схема отказоустойчивости

- #### NGINX
    - Можно использовать CARP: Один балансировщик active, другой standby. В случае выхода из строя active, standby займет его место
    - Проверяется доступность backend-сервисов. Если он не отвечает после retry запросов, то исключается из пула
- #### Kubernetess
    - Запросы распределяются только на живые поды, поды которые не проходят readiness probe исключаются из балансировки
    - При росте нагрузки добавляются новые поды

## 7. Алгоритмы

### 1) Алгоритм поиска товаров
1. После введенного запроса необходимо: провести токенизацию (разделить слова), привести все к нижнему регистру, отфильтровать стоп-слова, провести лемматизацию токенов. То есть задача - нормализовать входные данные. Этот этап позволяют реализовать внутренние инструменты ElasticSearch: `tokenizer`, фильтры `lowercase`, `russina_stop` и `russian_stemmer`.
2. Первый слой ранжирования. На этом этапе из всей базы товаров выбирается ограниченное множество наиболее релевантных кандидатов (несколько тысяч). Основной критерий — текстовая релевантность запроса, определяемая с помощью BM25 на основе проиндексированных текстовых полей в Elasticsearch.
Дополнительно при вычислении итогового ранга учитываются базовые поведенческие и популярностные признаки: количество заказов у товара, средний пользовательский рейтинг товара и число просмотров товара.
Результатом этого этапа является набор кандидатов, отсортированных по комбинированной метрике релевантности и популярности, которые далее передаются на второй слой ранжирования.
3. Второй слой ранжирования. На этом этапе нужно выполнить переупорядочивание кандидатов с учетом индивидуальных предпочтений пользователя: просмотры товаров, заказы и любимые категории. Для каждого кандидата вычисляется релевантность к пользователю.  
4. Итоговый рейтинг формируется как взвешенная комбинация текстовой релевантности и персонализации: <br> $Рейтинг = \alpha * текстовая\_релевантность + \beta * релевантность\_к\_пользователю$ <br> Учитывая, что в поиске все-таки важнее выдавать товары, похожие на запрос, то примем:
$\alpha = 0.8,\ \beta = 0.2$

### 2) Алгоритм рекомендаций
Необходимо показать пользователю товары, которые могут быть ему интересны, на основе:
- его поведения (просмотры, покупки)
- признаков товаров.
1. Сначала нужно векторизировать пользователя. Собираем его действия: последние просмотры, любимые категории, купленные товары. Каждому товару соответсвует `product_vector`, содержащий признаки (категория, цена, бренд). Формируем `user_vector` - средневзвешанное объединение `product_vector` товаров (среднее по каждой координате) с учетом важности действия (например, покупки имеют больший вес чем просмотры). Таким образом `user_vector` отображает предпочтения пользователя в многомерном признаковом пространстве.
2. Первый слой ранжирования. Определяем набор товаров, с которыми пользователь взаимодействовал за последнее время. Ищем товары аналогичну алгоритму поиска, но для каждого товара ищем сотни похожих, потом объединяем и убираем дупликаты.
3. Вторый слой ранжирования.Теперь нужно провести персонализацию кандидатов, полученных на втором этапе. Для этого вычисляем векторную близость `user_vector` и каждого `product_vector` с помощью косинусного расстояния.
4. В отличие от поиска персонализация должна иметь больший вес. Например: $\alpha = 0.3,\ \beta = 0.7$
